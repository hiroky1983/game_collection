# 思考ゲーム詰め合わせアプリ — MVP仕様書（仮称）

> iOS向け「考える系ゲーム」詰め合わせアプリ。MVPは **2048** と **将棋** の2本。
> オフライン完結・サーバーレス。実装はこの仕様を起点に進める。

---

## 1. コンセプト / スコープ方針

- ジャンル: **思考ゲーム詰め合わせ**（反射・カジュアル系は採用しない）
- 選定基準: 「AIに実装させた際、**ビジネスロジックの正しさを人間がレビュー・検証できるか**」を重視。決定的でテスタブルなゲームに絞る
- MVPの狙い: **最速で「遊べる詰め合わせ」を1本ストアに出し、器の設計と配信フローを検証する**。完成度より「出して回す」を優先
- フラッグシップは将棋。ただしMVPでは**簡易AIで割り切り**、本格AIはPhase 2へ

---

## 2. 対象プラットフォーム / 技術スタック

| 項目 | 採用 |
|---|---|
| プラットフォーム | **iOS のみ**（クロスプラットフォーム要件なし） |
| 言語 / UI | **Swift 6 系 / SwiftUI** |

- AdMob のアプリ登録・広告ユニットID設定
- アプリアイコン / スクリーンショット / ストアメタデータ

## 3. アーキテクチャ（拡張性の「継ぎ目」）

> MVPは2本だが、**後からゲームを足せる継ぎ目だけは最初に引く**。中身は薄くてよい。
> 遅延ロード前提なので、ゲーム本数が増えてもメモリ・起動時間は比例しない。

### 3.1 ゲーム登録プロトコル
```swift
protocol GameModule {
    var id: String { get }
    var title: String { get }
    var icon: Image { get }
    func makeView(services: GameServices) -> AnyView
}
```
- ハブ画面は登録された `GameModule` を**動的に列挙**するだけ
- 新ゲーム追加 = 1モジュール追加 + レジストリ登録で完結

### 3.2 横断サービス注入
- `GameServices` にスコア/広告/設定/永続化をまとめ、各ゲームに注入
- MVPでは中身は最小（広告・設定は空実装でも可）

### 3.3 将棋AIの境界（USI）
```swift
protocol ShogiEngine {
    func bestMove(sfen: String) async -> String   // "position → bestmove"
}
```
- MVPは `SimpleMinimaxEngine`（自前）を実装
- Phase 2で `YaneuraOuEngine`（C++ FFI）に**差し替えてもUI・ルールは無傷**

### 3.4 永続化ストア
```swift
protocol SnapshotStore {
    func save<T: Codable>(_ snapshot: T, for gameID: String) throws
    func load<T: Codable>(_ type: T.Type, for gameID: String) -> T?
    func clear(for gameID: String)
}
```
- `gameID`（`"shogi"` / `"2048"`）でキーを分離
- `load` の非nilで**ハブに「続きから」を表示**するか判定

## 4. スコープ（In / Out）

### MVPに入れる（In）
- 2048: 完全に遊べる（スコア・ゲームオーバー判定・スライドアニメ）
- 将棋: **対人（パス＆プレイ）＋ 簡易AI**
- ハブ画面（2本を選んで起動）
- バナー広告 1枚
- 対局/ゲームの**中断レジューム**
- 将棋の**終局後の検討**（手を戻す/進める）

### MVPに入れない（Out → Phase 2以降）
- 端末内の本格将棋AI（やねうら王統合）

- 棋譜履歴・対局統計の蓄積
- 詰将棋、チェス、リバーシ等の追加ゲーム
- Game Center / ランキング / 実績
- インタースティシャル広告、課金

## 5. 各ゲーム仕様

### 5.1 2048
- 盤面: 4×4、`0 = 空`
- 操作: スワイプ（上下左右）→ 寄せて隣接同値をマージ → 再度寄せる
- コア: **「1方向のスライド＆マージ」を純粋関数**として実装。残り3方向は盤の回転で使い回す
- タイル生成: 1手ごとに空きマスへランダム生成
- ゲームオーバー: 移動可能手なしで判定
- スコア: セッション中のみ表示（**ベストスコアは保存しない**）
- 中断: 盤面＋スコアをスナップショット保存。再開で復元、ゲームオーバーで破棄
- 検証: 状態遷移テーブルによるユニットテスト（決定的なので網羅可能）

### 5.2 将棋
- モード: 対人（同端末2人）＋ 簡易AI対局
- 簡易AI: **ミニマックス＋αβ＋簡易評価関数**（強さは問わない。「打てればOK」で深追いしない）
- AI境界: `ShogiEngine`（USI）。MVPは自前、Phase 2で本格エンジン差し替え

#### ルールエンジン要件
- 合法手生成は **pseudo-legal → legal** の2段階
  - pseudo-legal: 各駒の利き＋持ち駒打ち。**成・不成の両方を候補に出す**。行き所のない駒（最奥段の歩・香、最奥2段の桂）は不成を除外（強制成り）
  - legalフィルタ: 指した後に自玉が**王手放置にならない**手のみ
- 反則の実装（必須）: **二歩 / 行き所のない駒 / 打ち歩詰め / 王手放置**
- 千日手: MVPでは簡易検出 or 後回し（実戦頻度低）
- `makeMove` / `unmakeMove`（捕獲駒・成り状態をUndoに記録して完全復元）
  - perft・検討・AI探索の共通土台

#### 検証戦略（perft）
```
perft(0) = 1
perft(n) = Σ over legalMoves { make → perft(n-1) → unmake }
```
- **初期局面 depth1 = 30 / depth2 = 900**（確定値）をまず通す ← 最初の検収ライン
- depth3以降は公表値と照合（ak11氏「将棋でPerftしてみた」/ やねうら王mini サンプル）
- 最多合法手局面（593手、sfen公開あり）で単発テスト
- **最大の罠**: 「成・不成を別手として数えるか」。perft値を直接左右するので照合時に要注意

## 6. データモデル（永続化）

> 棋譜データ = **現在進行中の1局分だけ存在する一時データ**。常に0〜1件で積み上がらない。

### 将棋スナップショット
```swift
enum GamePhase: String, Codable { case playing, review }
enum PlayerKind: String, Codable { case human, ai }

struct ShogiSnapshot: Codable {
    var initialSfen: String      // 開始局面（平手の標準SFEN）
    var moves: [String]          // USI形式の指し手列 ["7g7f", "3c3d", "P*5e", "8h2b+"]
    var phase: GamePhase         // .playing / .review
    var reviewPly: Int?          // 検討で表示中の手数（playing時は nil = 末尾）
    var sente: PlayerKind        // 先手が人間/AI
    var gote: PlayerKind         // 後手が人間/AI
    var aiLevel: Int?            // 簡易AIの強さ（人間同士なら nil）
    var startedAt: Date          // KIFヘッダ・整理用
}
```
- `moves` を **USI文字列で保持**（USI境界・KIF生成・棋譜再生のすべてに接続）
- 手番は `moves.count` の偶奇で導出（冗長に持たない）
- この1データで**中断復帰・検討・KIFエクスポート**の3用途を賄う

### 2048スナップショット
```swift
struct Game2048Snapshot: Codable {
    var board: [[Int]]   // 4×4、0 = 空
    var score: Int       // 現在スコア（bestScore は持たない）
}
```
- 盤面そのものを保存するため乱数シードの再現は不要

### ライフサイクル
```
対局開始 → (指し手追記、中断時 save)
        → 終局 → 検討（手を戻す/進める）＋ KIFエクスポート可
        → 対局破棄 / 検討終了 → clear（スナップショット削除・メモリ解放）
```

---

## 7. 棋譜の扱い

- 役割は3つ: **中断レジューム / 終局後の検討 / KIFエクスポート**
- 内部表現は `moves`（USI）。**KIF（`.kif`）はエクスポート時に生成変換**
- エクスポートUI: iOS の `ShareLink`（Share Sheet）でテキスト共有 → メモ/ファイル/LINE等へ
- **アプリ内には貯めない**（外に出すだけ）

---

## 8. 通信・永続化の方針（重要）

- **リモート（サーバー）への保存は一切なし**（オフライン / サーバーレス）
- ローカル永続化は**中断スナップショット1件のみ**（常に上書き、積み上がらない）
- **ベストスコア・棋譜履歴・統計は保持しない**
- **広告はオンライン時のみ表示**。広告のロード失敗は握りつぶし、**ゲーム本体はオフラインで通常動作**させる
- 端末をまたぐ同期・サーバー集計ランキング等が必要な機能はすべて Phase 2 以降

## 9. 実装マイルストーン / DoD

| # | 内容 | DoD |
|---|---|---|
| **M1** | 器の足場（Core: GameModule/GameServices/SnapshotStore + ハブ + SPM分割）に2048を1本登録 | ハブから2048が開く |
| **M2** | 2048を完成（スコア・ゲームオーバー・スライドアニメ・中断復帰） | 単体で遊べる + 状態遷移テスト緑 |
| **M3** | 将棋ルールエンジン + 対人 | 反則なく1局指せる + **perft 30/900 一致** |
| **M4** | 将棋に簡易AI（USI境界の裏にミニマックス） | 一人で最後まで打てる（強さ不問） |
| **M5** | バナー広告 + 審査準備（ATT/プライバシーマニフェスト/アイコン/スクショ） | TestFlight → 申請 |

> 最大の不確実性は **M3（将棋ルールエンジン）**。perft照合を最初から組み込みバグを早期に潰す。
> M2の時点で「動く詰め合わせ」が手元にある状態になる。

## 10. Phase 2 以降のロードマップ（参考）

- **v1.1**: 将棋の端末内本格AI（やねうら王 C++ を Objective-C++ ブリッジで統合、NNUE評価関数は **On-Demand Resources / 初回DL** で本体から分離）
- **v1.2**: 詰将棋（問題データ同梱・日替わり出題）＋ Game Center（ランキング・実績）
- **v1.3 以降**: チェス（chess.dart相当 + Stockfish）・リバーシ等を同じ `GameModule` テンプレに流し込む

### スケール時の注意（先回りメモ）
- ハブ→各ゲームは**遅延ロード**を徹底（メモリ・起動を本数に依存させない）
- 重いアセット（NNUE評価関数・問題集）は**On-Demand Resources で本体DLサイズから分離**
- 駒画像・効果音・盤テクスチャは**共通モジュールで共有**して重複を排除
- 純ロジック系ゲームのコードサイズは無視できるので本数を気にせず追加可
